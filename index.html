üìñ Chapter 1: Introduction to Software Engineering

What is Software Engineering?

Definition: Software engineering is an engineering discipline that is concerned with all aspects of software production from the early stages of system specification through to maintaining the system after it has gone into use.

Essential Attributes of Good Software

üéØ Maintainability

Software should be written in such a way that it can evolve to meet the changing needs of customers.

‚ö° Dependability & Security

Software must be trustworthy including reliability, security, and safety.

üöÄ Efficiency

Software should not make wasteful use of system resources such as memory and processor cycles.

‚úÖ Acceptability

Software must be acceptable to the users it was designed for. It must be understandable, usable, and compatible.

Software Process Activities

  1. Software Specification: Defining what the system should do
  2. Software Development: Designing and programming the system
  3. Software Validation: Checking that it does what the customer wants
  4. Software Evolution: Changing the system in response to changing customer needs

Types of Software Products

üì¶ Generic Products

  • Stand-alone systems sold to any customer
  • Specification controlled by developer
  • Examples: PC software, mobile apps

üéØ Customized Products

  • Commissioned by a particular customer
  • Specification controlled by customer
  • Examples: Embedded systems, business software

General Issues Affecting Software

  • Heterogeneity: Systems operate on different platforms
  • Business & Social Change: Rapid changes require fast adaptation
  • Security & Trust: Critical as software is integrated into all aspects of life
  • Scale: From small embedded systems to cloud-based systems

Software Engineering Ethics

ACM/IEEE Code of Ethics Principles:

  1. Act in the PUBLIC interest
  2. Act in the best interests of CLIENT and EMPLOYER
  3. Ensure PRODUCT meets highest professional standards
  4. Maintain INTEGRITY and INDEPENDENCE
  5. Promote ethical approach to MANAGEMENT
  6. Advance integrity and reputation of the PROFESSION
  7. Be fair and supportive of COLLEAGUES
  8. Participate in LIFELONG LEARNING

üîÑ Chapter 2: Software Processes

The Software Process

A software process is a set of related activities that leads to the production of a software product. There are many different processes, but all must include four fundamental activities:

  • Software Specification
  • Software Design and Implementation
  • Software Validation
  • Software Evolution

Plan-Driven vs. Agile Processes

üìã Plan-Driven

  • All activities planned in advance
  • Progress measured against plan
  • Works well for large systems
  • Heavy documentation

‚ö° Agile

  • Planning is incremental
  • Easier to change process to reflect requirements
  • Works well for small/medium systems
  • Minimal documentation

Software Process Models

1. The Waterfall Model

Plan-driven model with separate and distinct phases of specification and development.

Requirements Definition ‚Üí System & Software Design ‚Üí Implementation & Unit Testing ‚Üí Integration & System Testing ‚Üí Operation & Maintenance
Advantages:
  • Simple and easy to understand
  • Works well for small projects with clear requirements
  • Processes and results well documented
Disadvantages:
  • Difficult to accommodate change after process is underway
  • Few systems have stable requirements
  • Customer must be patient - working version available late

2. Incremental Development

System developed and delivered in increments. May be plan-driven or agile.

Benefits:
  • Cost of accommodating changes is reduced
  • Easier to get customer feedback
  • More rapid delivery and deployment
Problems:
  • Process is not visible
  • System structure tends to degrade
  • Requires good tools and environment

3. Integration and Configuration (Reuse-oriented)

System assembled from existing components and COTS (Commercial Off-The-Shelf) systems.

Types of Reusable Software:
  • Stand-alone application systems
  • Collections of objects (classes, packages)
  • Web services
Key Stages:
  1. Requirements specification
  2. Software discovery and evaluation
  3. Requirements refinement
  4. Application system configuration
  5. Component adaptation and integration

Software Testing

Component Testing

Testing individual components independently. Also known as unit testing.

System Testing

Testing the system as a whole, checking emergent properties.

Acceptance Testing

Testing with customer data to check system meets customer needs.

Coping with Change

System Prototyping:

A version of the system is developed quickly to check customer requirements and design decisions.

Benefits:
  • Improved system usability
  • Closer match to user needs
  • Improved design quality
  • Improved maintainability
  • Reduced development effort

Incremental Delivery:

Develop the system in increments and evaluate each increment before proceeding to the next. User requirements prioritized and highest priority requirements included in early increments.

Advantages:
  • Customer value delivered with each increment
  • Early increments act as prototype
  • Lower risk of overall project failure
  • Highest priority services receive most testing

üìã Chapter 4: Requirements Engineering

Types of Requirements

Functional Requirements

Statements of services the system should provide, how the system should react to inputs, and how it should behave in particular situations.

Non-Functional Requirements

Constraints on services or functions offered by the system (timing, development process, standards, etc.)

User Requirements

Written for customers, stating what services will be provided in natural language

System Requirements

Detailed descriptions of the system's functions, services, and operational constraints

Requirements Engineering Process

1. Requirements Elicitation ‚Üí 2. Requirements Analysis ‚Üí 3. Requirements Validation ‚Üí 4. Requirements Management

Requirements Elicitation Techniques

üí¨ Interviews

Description: Formal or informal discussions with stakeholders

Best For: Understanding workflows and problems

üë• Ethnography

Description: Observing users in their work environment

Best For: Social and organizational requirements

üìñ Stories & Scenarios

Description: Real-life examples of how system is used

Best For: Developing use cases

üé® Prototyping

Description: Creating executable models

Best For: Validating requirements

Requirements Validation

Checking that requirements define the system the customer really wants.

Validation Checks:
  • Validity: Does the system provide functions supporting customer needs?
  • Consistency: Are there conflicting requirements?
  • Completeness: Are all functions required by customer included?
  • Realism: Can requirements be implemented given budget and technology?
  • Verifiability: Can requirements be checked?

Requirements Management

Requirements inevitably change due to:

  • Business and technical environment changes
  • Different stakeholder perspectives
  • Improved understanding of requirements

Solution: Requirements management planning including version control, change management, and traceability policies.

Requirements Specification

Ways to Write Requirements:

  1. Natural Language: Simple but can be ambiguous
  2. Structured Natural Language: Uses templates to reduce ambiguity
  3. Design Description Languages: More formal notation
  4. Graphical Notations: Use case diagrams, sequence diagrams
  5. Mathematical Specifications: Formal methods for critical systems

üèÉ Chapter 3: Agile Software Development

Agile Manifesto Values

  1. Individuals and interactions over processes and tools
  2. Working software over comprehensive documentation
  3. Customer collaboration over contract negotiation
  4. Responding to change over following a plan

Agile Principles

üéØ Customer Involvement

Customers should be closely involved throughout development

üì¶ Incremental Delivery

Software developed in increments with customer specifying requirements

üë• People not Process

Skills of team should be recognized and exploited

üîÑ Embrace Change

Expect requirements to change and design to accommodate

‚ú® Maintain Simplicity

Focus on simplicity in both software and development process

Extreme Programming (XP)

Perhaps the best-known and most widely used agile method.

Key Practices:

  • Incremental Planning: Requirements as user stories on cards
  • Small Releases: Frequent releases of system
  • Simple Design: Just enough design for current requirements
  • Test-First Development: Write tests before code
  • Refactoring: Continuously improve code structure
  • Pair Programming: Developers work in pairs
  • Collective Ownership: Anyone can change any code
  • Continuous Integration: Integrate and test frequently
  • Sustainable Pace: No excessive overtime
  • On-site Customer: Full-time customer representative

Scrum

Agile method focusing on managing iterative development.

Key Roles:

  • Product Owner: Represents stakeholders
  • Scrum Master: Ensures process is followed
  • Development Team: Cross-functional team (5-9 people)

Key Artifacts:

  • Product Backlog: Prioritized list of requirements
  • Sprint Backlog: Tasks for current sprint
  • Increment: Potentially shippable product

Key Events:

  • Sprint: 2-4 week time-boxed iteration
  • Sprint Planning: Team selects items from product backlog
  • Daily Scrum: 15-minute stand-up meeting
  • Sprint Review: Demonstrate increment to stakeholders
  • Sprint Retrospective: Team reflects on process

When to Use Agile

Best Suited For:

  • Product development where software is updated frequently
  • Custom system development within an organization
  • Small to medium-sized products
  • Projects with unclear or changing requirements

Challenges:

  • Difficult to maintain involvement with all stakeholders
  • Team members may not have suitable personality for intense involvement
  • Prioritizing changes can be difficult
  • Maintaining simplicity requires extra work
  • Contracts may prefer plan-driven approaches

üìù User Stories & Examples

What are User Stories?

User stories are short, simple descriptions of a feature told from the perspective of the person who desires the new capability.

Template: As a [type of user], I want [goal] so that [reason].

INVEST Criteria for Good User Stories

I - Independent

Should be self-contained

N - Negotiable

Details can be discussed

V - Valuable

Must deliver value to user

E - Estimable

Must be able to estimate size

S - Small

Should fit in one iteration

T - Testable

Must have acceptance criteria

Example User Stories

E-Commerce System: 1. As a customer, I want to add items to my shopping cart so that I can purchase multiple items at once. 2. As a customer, I want to save items for later so that I can purchase them when I'm ready. 3. As an admin, I want to view sales reports so that I can track business performance. 4. As a customer, I want to receive email confirmation so that I know my order was placed successfully. 5. As a customer, I want to track my order so that I know when to expect delivery.
Social Media Platform: 1. As a user, I want to create a profile so that I can share information about myself. 2. As a user, I want to post updates so that my friends can see what I'm doing. 3. As a user, I want to like and comment on posts so that I can interact with my friends. 4. As a user, I want to control my privacy settings so that I can decide who sees my content. 5. As a user, I want to report inappropriate content so that the platform remains safe.
Banking Application: 1. As a customer, I want to check my balance so that I know how much money I have. 2. As a customer, I want to transfer money between accounts so that I can manage my finances. 3. As a customer, I want to receive notifications for transactions so that I can detect fraud. 4. As a customer, I want to pay bills online so that I don't have to visit the bank. 5. As a customer, I want to set up automatic payments so that I don't miss due dates.
